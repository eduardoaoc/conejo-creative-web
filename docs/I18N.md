# I18N — Internacionalización

## Idiomas

| Locale | Idioma                  | Variante de contenido | `lang` / hreflang |
| ------ | ----------------------- | --------------------- | ----------------- |
| `es`   | Español (**principal**) | Español de España     | `es`              |
| `pt`   | Portugués               | Portugués de Brasil   | `pt-BR`           |
| `en`   | Inglés                  | Inglés internacional  | `en`              |

- `/` redirige **siempre** a `/es` (detección por `Accept-Language` desactivada a propósito: `localeDetection: false` en `src/i18n/routing.ts`).
- Todas las páginas existen en los tres idiomas (SSG por locale).
- Los segmentos de URL permanecen en español (`/servicios`, `/nosotros`…) en los tres idiomas en esta etapa.

## Piezas de next-intl

| Fichero                  | Función                                                    |
| ------------------------ | ---------------------------------------------------------- |
| `src/i18n/routing.ts`    | Locales, locale por defecto, prefijo siempre visible       |
| `src/i18n/request.ts`    | Carga los mensajes del locale de la petición + `timeZone`  |
| `src/i18n/navigation.ts` | `Link`, `useRouter`, `usePathname`… conscientes del idioma |
| `src/proxy.ts`           | Redirecciones (`/` → `/es`) y negociación de rutas         |
| `next.config.ts`         | `createNextIntlPlugin('./src/i18n/request.ts')`            |

**Regla de oro**: importar `Link`, `useRouter`, `usePathname` y `redirect` **siempre** desde `@/i18n/navigation`, nunca desde `next/link` o `next/navigation` (salvo APIs sin equivalente, como `notFound`).

## Mensajes

```text
messages/
├── es.json   ← fuente de verdad (idioma principal)
├── pt.json
└── en.json
```

- **Ninguna string visible se escribe en componentes.** Excepción única: el nombre propio «ConejoCreative», que es invariante.
- Los tres ficheros comparten estructura idéntica. Al añadir una clave, añádela en los tres.
- Espacios de nombres actuales: `meta`, `layout`, `nav`, `localeSwitcher`, `footer`, `home`, `services`, `projects`, `about`, `contact`.

### Crear una clave nueva

1. Añadir la clave en `messages/es.json` dentro del namespace correspondiente.
2. Replicarla en `pt.json` y `en.json` con su traducción.
3. Consumirla:
   - Server Component síncrono: `const t = useTranslations('namespace')`
   - Server Component asíncrono / metadata: `const t = await getTranslations('namespace')`
   - Client Component: `useTranslations` (los mensajes llegan por `NextIntlClientProvider`).

### Crear un idioma nuevo

1. Añadir el locale a `locales` en `src/i18n/routing.ts`.
2. Crear `messages/<locale>.json` (copiar la estructura de `es.json`).
3. Añadir el nombre del idioma bajo `localeSwitcher` en **todos** los ficheros de mensajes.
4. Mapear su etiqueta BCP 47 en `localeLangTags` y `openGraphLocales` (`src/config/site.ts`).
5. Listo: rutas, sitemap, hreflang, selector y tests paramétricos lo recogen de `routing.locales`.

## Selector de idioma

`components/navigation/LocaleSwitcher.tsx` — `<select>` nativo (accesible por defecto). Usa `usePathname()` de `@/i18n/navigation`, que devuelve la ruta **sin** prefijo de idioma, y `router.replace(pathname, { locale })`: por eso la ruta actual se conserva al cambiar de idioma.

## SEO internacional

Ver docs/SEO.md. En resumen: canonical por idioma, `hreflang` (`es`, `pt-BR`, `en`, `x-default` → español) generados en `lib/metadata.ts`, sitemap con `alternates` por URL, y `<html lang>` correcto por locale.
